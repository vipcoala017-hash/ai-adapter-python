# 复验规则

## 测试

### 规则规则
程序执行前先规划测试用例。交付前必须执行本阶段相关测试，并形成或更新测试报告。

### 测试覆盖

每轮测试前先列本轮高风险项，再确认是否已有查询字段和测试用例。

### 高风险测试内容

- 短窗口动作要按关键时序拆分测试，不写笼统用例。
- 同一动作在不同触发时序下结果不同，必须拆成独立用例。
- 动作名称相近但语义不同，必须拆分命名和断言。
- 同类对象变体至少覆盖一个非默认对象。
- 涉及视觉资源时，查询接口不能只校验元素可见或节点存在，还要校验贴图或资源已加载。
- 涉及数值或配置变更时，通过查询增加明确断言，不能只看动作是否进入某状态。

### Bug 治理

发现 bug 后补录 bug 单。bug 单最少字段：

- bug 编号
- 发现时间
- 发现来源
- 所属阶段
- 问题现象
- 事实依据
- 根因归类
- 修复动作
- 预防动作
- 是否已回写约束
- 当前状态

归因必须基于代码、日志、配置或文档事实，不只写“已修复”。

## 验收

### 验收原则

每个程序产出后都必须有验收视角复核。验收不通过时，必须修复、补齐内容。

### 复核依据

只基于事实复核：

- 用户需求
- 仓库文档
- 代码实现
- 配置与资源
- 编译结果
- 测试结果

不允许因为“看起来对了”判定通过。必须复核以下内容：

- 是否满足用户需求。
- 是否满足本轮影响领域的规则。
- 是否满足接口文档或说明。
- 是否存在阻塞下一阶段的问题。
- 是否存在旧验收结论失效。
- 是否存在表面通过但关键细节未核实。
- 是否存在测试通过但覆盖不足。
- 是否存在未更新的文档、测试或数据。
- 是否存在临时文件、密钥、无关格式化或无关重构。

### 程序专项验收

接口验收检查：

- 模块边界是否清晰。
- 对外接口是否稳定、可实现、可测试。
- 接口文档是否完整。
- 调用约束、数据影响、错误处理是否明确。

代码验收检查：

- 代码实现是否符合已通过接口。
- 函数注释是否准确。
- 测试文档是否同步。
- 编译或静态检查是否通过。
- 是否出现接口漂移。

### 验收输出

内部验收结论写入验收报告，至少包含：

- 验收结论
- 是否通过
- 不通过或失效原因
- 必须补齐项
- 可后补项
