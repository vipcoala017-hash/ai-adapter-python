# 知识沉淀规则

## 目标

知识沉淀用于把一次开发、排障、验收、用户反馈中得到的稳定经验转化为可复用规则，降低后续重复错误。沉淀内容必须基于事实，不把一次性猜测、项目私有实现细节或尚未验证的偏好写成通用规则。

本规则面向单体程序开发 Agent。原 `.codex` 中的 `skill` / `agent` 回写思想，在这里转换为当前目录下的规则文档维护和项目内质量文档维护。

## 沉淀触发时机

出现以下情况时必须检查是否需要沉淀：

1. 修复 bug 后发现可复用的预防规则。
2. 用户反馈暴露了测试漏检、验收漏检或需求理解偏差。
3. CLI、自动化测试、运行日志暴露了稳定的高风险模式。
4. 同类问题第二次出现。
5. 新增或调整了跨模块接口、目录边界、状态流转、资源加载、数据读取、UI 交互或平台封装方式。
6. 发现现有规则过宽、过窄、含糊、过期或互相冲突。
7. PR 交付后有无法本轮修复但必须长期跟踪的风险。

## 事实要求

任何沉淀项都必须能追溯到事实依据。允许的事实来源：

- 代码事实：类、函数、接口、调用链、状态变更、资源引用。
- 文档事实：接口文档、设计文档、测试报告、CLI 文档、PR 描述。
- 测试事实：测试用例、CLI 查询结果、CI 结果、Unity 编译或运行日志。
- 资源事实：文件存在、导入配置、加载路径、图集关系、贴图加载结果。
- 用户反馈事实：用户指出的问题、复现路径、期望行为。

禁止把“看起来应该”“一般项目会”“模型推测”写成规则。

## 沉淀分层

沉淀前先判断内容应该写到哪里。

### 当前 Agent 规则层

通用、稳定、可跨项目复用的规则写入当前目录 `rules/`。

- `rules/core_principles.md`：协作基线、事实优先、自主执行、仓库保护、范围控制。
- `rules/program_rules.md`：程序分层、命名、模块职责、接口边界。
- `rules/comment_rules.md`：AI 友好注释、注释检索标签、注释查询方式。
- `rules/resource_rules.md`：资源接入、资源命名、加载边界、资源验收。
- `rules/log_rules.md`：日志、运行态事实、排障记录。
- `rules/knowledge_rules.md`：知识沉淀、规则回写、经验分层。
- `rules/04-testing-cli-quality.md`：测试、CLI、漏检、bug 台账和质量治理。
- `rules/05-review-acceptance.md`：内部验收、状态失效、验收漏检和打回规则。

### 项目文档层

只对当前项目成立的事实、设计、约束、风险和 bug，写入目标项目的 `docs/`。

推荐落点：

- `docs/quality/bug-ledger.md`：bug 单、漏检、根因、修复动作。
- `docs/quality/bug-review.md`：周期复盘、高频错误、待跟踪项。
- `docs/testing/test-cases.md`：测试用例。
- `docs/testing/test-report.md`：测试报告。
- `docs/cli/cli-reference.md`：CLI 操作入口和查询入口。
- `docs/design/program/interfaces/`：接口契约。
- `docs/design/program/module-design.md`：模块设计和边界。

### PR 交付层

本轮发现但不适合写入规则或项目文档的内容，写入 PR 描述的风险与后续项。PR 层内容不能替代 bug 台账或规则沉淀。

## 通用规则与项目私有经验的判定

写入 `rules/` 前必须通过以下检查：

- 是否能在其他游戏或程序项目中复用。
- 是否描述的是流程、约束、检查项或稳定模式，而不是某个文件的临时实现。
- 是否已经被至少一次事实验证。
- 是否不会与当前已有规则冲突。
- 是否足够具体，可以指导后续 Agent 执行。

不满足这些条件时，不写入通用规则。只写入项目文档、bug 台账或 PR 风险。

## Bug 沉淀规则

每次发现 bug 后，必须补录 bug 单。不能只修代码不记录原因。

bug 单最少字段：

- bug 编号
- 发现时间
- 发现来源
- 所属阶段
- 问题现象
- 复现路径
- 事实依据
- 根因归类
- 修复动作
- 新增或更新的测试
- 预防动作
- 是否已沉淀为规则
- 当前状态

根因必须基于事实，至少引用代码、日志、配置、文档、CLI 查询或用户反馈中的一种。

## 漏检沉淀规则

如果问题属于当前阶段已宣称通过的范围，却由用户反馈、运行态事实或后续回归暴露，必须先判定为漏检。

### 测试漏检

触发条件：

- 功能已声明测试通过，但问题属于测试范围。
- 现有测试用例存在但断言过宽。
- 缺少查询字段、缺少对象变体、缺少失败路径或缺少时序路径。

必须沉淀：

- 漏检原因。
- 缺失用例或缺失断言。
- 新增查询字段或 CLI 入口。
- 受影响范围重测结果。
- 是否需要新增测试规则。

### 验收漏检

触发条件：

- 代码、接口、UI、资源、平台、数据或测试已验收通过，但后续事实否定该结论。
- 验收只看“测试通过”，没有检查测试充分性。
- 验收未核对关键字段、状态、边界、约束、交互或验收条件。

必须沉淀：

- 漏检所属验收阶段。
- 本应发现但未发现的原因。
- 需要补强的验收项。
- 新增回归用例。
- 原验收结论失效范围。

## 高频错误升级规则

同类错误累计出现时，必须聚类统计并判断是否升级为新规则。

升级条件：

- 同一根因重复出现两次及以上。
- 单次问题影响范围大，可能反复导致 PR 失败、测试漏检、接口漂移或用户可见错误。
- 问题无法仅靠本地修复长期预防。

升级后必须写清：

- 新规则标题。
- 触发场景。
- 禁止事项。
- 必须执行的检查项。
- 推荐的验证方式。
- 适用范围。
- 不适用范围。

## 规则回写流程

沉淀规则时按以下顺序执行：

1. 确认事实来源。
2. 判断沉淀层级：通用规则、项目文档、bug 台账、PR 风险。
3. 检查是否已有同类规则，优先补充已有规则，不重复造新标题。
4. 将经验改写成可执行约束，不写流水账。
5. 补充反例或禁止事项，避免规则被宽泛理解。
6. 若规则影响测试或验收，同步更新测试规则或验收规则。
7. 若规则影响注释检索，同步更新 `rules/comment_rules.md` 的标签或查询说明。
8. 更新后复查编码、可读性和链接路径。

## 可执行规则写法

好的沉淀规则应包含：

- 适用场景
- 必须做什么
- 禁止做什么
- 判断依据
- 验证方式
- 失败后的处理动作

不合格写法：

- “以后注意这个问题”
- “优化测试”
- “加强检查”
- “这里容易错”
- “看情况处理”

合格写法示例：

```markdown
### 短窗口状态测试

适用场景：动作状态只持续少量帧，或状态可能在同帧被清理逻辑取消。

必须执行：

- 提供可稳定采样的 CLI 查询字段。
- 测试至少覆盖最早触发时机、稳定触发时机和高风险边界时机。
- 查询前不得插入无关等待步骤。

禁止：

- 禁止只通过日志或人眼观察判断状态出现过。
- 禁止用一个宽泛断言覆盖多个不同时序分支。

验证方式：

- 执行定向 CLI 测试，并在测试报告中记录采样点和查询结果。
```

## PR 前知识检查

创建 PR 前必须检查：

- 本轮是否出现 bug、测试漏检或验收漏检。
- 本轮是否发现可复用规则。
- 本轮是否修改了接口、CLI、测试、数据、资源或注释规则。
- 本轮是否有项目私有经验需要写入 `docs/quality/`。
- 本轮是否有通用经验需要写入 `rules/`。
- PR 描述是否列出了未沉淀但需要后续处理的风险。

如果答案为“是”，必须完成对应沉淀或说明不沉淀原因。

## 禁止事项

1. 禁止把项目私有事实写成通用规则。
2. 禁止把未经验证的推测写成规则。
3. 禁止只记录“已修复”，不记录根因和预防动作。
4. 禁止发现漏检后只补代码，不补测试或验收规则。
5. 禁止把新规则堆到总控文档，导致规则不可检索。
6. 禁止重复创建同义规则。
7. 禁止规则写得过宽，无法指导具体执行。
8. 禁止沉淀后不复查文档编码和可读性。
