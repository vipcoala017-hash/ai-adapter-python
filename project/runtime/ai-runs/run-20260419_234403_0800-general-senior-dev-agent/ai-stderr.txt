# 程序开发规则

## 架构

### 结构分层
`core / module / system / manager / object / scheduler / operation / ui / utils / data / config`。

各分层、module需严格遵循“单一职责”原则，职责边界清晰，无交叉、无冗余

### Core
核心层。基础骨架，负责保障程序稳定运行，是所有分层的核心支撑。是程序启动、运行、终止的核心入口。
#### 规范要求
1. 需包含全局异常处理机制，避免程序直接崩溃，需输出明确的异常日志。
3. 需预留扩展接口，支持新增module模块的无缝扩展，不影响原有核心逻辑。
4. 负责全局统一管理各分层、各module的实例，确保调用链路清晰。
#### 绝对禁止
1. 禁止遗漏全局异常处理逻辑，禁止程序因未捕获异常直接崩溃；
2. 禁止与其他分层（尤其是module、system层）出现功能重叠、职责交叉，不承担具体业务逻辑或模块实现；
3. 禁止修改核心层基础逻辑后，未同步更新依赖的其他分层。

### Module
模块层。功能组件载体，封装独立的业务子模块，实现功能模块化拆分，便于维护、扩展和复用。
#### 规范要求
1. 每个module需实现单一功能，边界清晰。
2. 遵循统一访问规范，确保可被调度层调度。
3. 模块间需通过调度层进行通信，新增、修改模块不影响其他模块。
4. 需包含自身的异常处理逻辑，模块内错误不扩散至其他模块。
5. 新功能接入优先封装为独立module，不回改已有module本体。
#### 绝对禁止
1. 禁止一个module承担多个无关功能。
2. 禁止模块间直接调用、耦合，禁止修改一个module导致其他模块异常。
3. 禁止module越权处理非当前module的职责。
4. 禁止模块内无异常处理逻辑。
5. 禁止新增功能时，不封装为module，直接非module层堆叠代码。

### System
系统层。负责承载程序核心逻辑、业务规则定义，统筹各module的协同工作。
#### 规范要求
1. 仅完成业务核心逻辑
2. 需包含业务规则校验逻辑
3. 业务逻辑需清晰可追溯，配合manager下Object对象层完成对象协同。
#### 绝对禁止
1. 禁止与其他层出现功能冲突。
2. 禁止在system层重复实现module已有的功能，禁止直接修改module内部逻辑。
3. 禁止直接接收未经operation层转发的操作指令，禁止越权管理manager下Object对象层的对象实例。

### Manager
管理器。Object生命周期管理，提供统一的对象访问入口，只维护对象的稳定抽象、生命周期。
#### 规范要求
1. manager下Object对象层的生命周期管控，接收调度层的生命周期协同请求。
2. 精准维护所有实例的生命周期状态，确保资源合理分配，无资源泄露，同步生命周期状态至独立调度层、manager下Object对象层；
3. 新对象（module、Agent、Object实例、调度实例）接入时，负责完成实例初始化、生命周期注册，优先通过接口、注册点接入，不回改manager本体，保持manager逻辑稳定；
4. 包含生命周期异常处理逻辑（如实例启动失败、异常终止），及时反馈至独立调度层，由独立调度层统筹处理，不干预调度流程和业务逻辑；
5. 逻辑需简洁清晰，不包含具体业务规则、调度逻辑、对象管理细节，仅专注于生命周期管控，与独立调度层、manager下Object对象层保持接口协同。
6. 需实现对象实例复用机制，避免重复创建对象导致资源浪费。
#### 绝对禁止
1. 禁止manager层集中堆叠具体业务逻辑。
2. 禁止越权处理manager下Object对象层的具体对象操作
2. 禁止越权处理其他层的业务规则，禁止与其他分层职责交叉。
2. 新对象接入时，禁止破坏manager的抽象稳定性。
3. 禁止调度逻辑混乱，导致业务流程断裂。
5. 禁止生命周期管理混乱，禁止出现实例冗余、资源泄露、生命周期状态记录错误的情况。
6. 禁止直接接收未经operation层转发的操作指令。

### Object
对象，隶属于manager层。每一类对象都统一由特定管理器管理。
#### 规范要求
1. 仅接受manager层的生命周期管控指令。
2. 配合manager层完成对象的启停、复用，确保对象实例无冗余、无泄露。
3. 包含对象异常处理逻辑，及时反馈至manager层，不自行处理异常或干预其他分层.
#### 绝对禁止
1. 禁止Object对象层脱离manager层独立运行，禁止不接受manager层的生命周期管控指令、独立调度层的对象调用指令，自行管理对象实例
2. 禁止Object对象层包含业务逻辑、调度分发、操作转发、数据处理、UI展示等功能，禁止越权处理system层、module层、独立调度层的职责
4. 禁止新对象实例直接接入Object对象层，必须通过manager层注册点接入，禁止未校验就完成对象初始化
5. 禁止Object对象层自行处理对象异常，禁止异常未反馈至manager层，禁止干预manager层的生命周期管理、独立调度层的调度决策
6. 禁止Object对象层越权执行调度功能，禁止替代独立调度层分发指令，禁止直接操作业务数据或module内部逻辑

### Scheduler
调度层。负责所有分层、各module、manager下Object对象层的协同调度、指令分发，承接operation层转发的用户操作指令。
#### 规范要求
1. 仅负责调度分发，确保业务流程连贯
2. 与其他层保持协同，接收其他层的状态变化同步
3. 调度逻辑需清晰可追溯，便于问题排查
4. 包含调度异常处理逻辑
5. 需适配不同业务场景的调度需求，确保调度高效、无卡顿。
#### 绝对禁止
1. 禁止调度层承担生命周期管理、对象管理具体操作、业务逻辑实现，禁止越权处理业务
2. 禁止调度逻辑混乱，导致业务流程断裂
3. 禁止独立调度层直接接收UI层传递的用户操作指令，必须通过operation层转发
4. 禁止未查询对象状态就进行调度
5. 禁止调度日志缺失、不可追溯，禁止调度异常未反馈至相关分层
6. 禁止调度层其他分层职责交叉

### Operation
作为用户操作与其他层的操作转发枢纽，专门负责接传递用户操作指令，进行指令校验、格式标准化、转发分发，不参与业务逻辑实现、不处理数据存储，核心职责是规范用户操作指令的传递链路，确保指令传递准确、可追溯，是用户与逻辑交互的唯一桥梁。
#### 规范要求
1. 仅接收用户操作指令，不主动发起操作
2. 需对接收的用户操作指令进行校验，过滤无效指令、非法操作，输出指令校验日志
3. 对合法指令进行格式标准化处理，确保指令格式与独立调度层的接口要求一致，便于后续分层处理
4. 按指令类型，将标准化后的指令精准转发至独立调度层，由独立调度层进行后续分发
5. 需与UI层、调度层保持接口兼容，支持指令类型的扩展，不影响原有指令转发逻辑
6. 包含指令传递异常处理逻辑，及时反馈，不导致程序异常
#### 绝对禁止
1. 禁止主动发起操作指令
2. 禁止operation层包含任何业务逻辑、数据处理、module实现、UI展示功能，禁止越权处理其他层的内容
3. 禁止不对用户操作指令进行校验，直接转发无效指令、非法操作至其他分层
4. 禁止指令格式未标准化就转发
5. 禁止operation层直接调用module层、core层、platform层、manager层、manager下Object对象层接口，禁止干预其他分层的正常运行

### UI
负责程序的可视化展示、交互操作，仅承担 “展示 + 交互” 功能，不参与业务逻辑实现。
#### 规范要求
1. 界面布局清晰、简洁，需适配不同尺寸的显示需求，无布局错乱
2. 交互逻辑简洁直观，需包含操作反馈和结果反馈
3. 所有用户操作指令必须通过operation层提供的接口传递
4. 动画反馈需简洁高效，不影响程序运行性能，不出现卡顿、延迟，反馈内容与operation层、独立调度层转发的操作结果一致
#### 绝对禁止
1. 禁止UI界面出现布局混乱、操作卡顿、渲染异常的情况
2. 禁止UI层包含任何业务逻辑、数据处理、module调用等非展示类功能
3. 禁止因UI设计不当导致操作失误、指令传递失败，禁止UI层直接修改程序配置或数据
4. 禁止UI层越权调用，不经过operation层直接传递用户操作指令
5. 禁止UI层包含复杂规则判断

### Utils
负责程序的辅助功能，提供通用工具类、工具方法。
#### 规范要求
1. 工具类、工具方法需通用可复用，不绑定具体业务逻辑
5. 工具方法需简洁高效，不占用过多系统资源，不影响程序运行性能
#### 绝对禁止
1. 禁止工具层承担业务逻辑等功能；
2. 禁止工具层出现功能冗余、运行卡顿，禁止工具方法占用过多系统资源
3. 禁止工具层干预其他分层的正常运行
4. 禁止工具方法与具体业务绑定，禁止无法复用的工具类存在
5. 禁止工具方法出现异常未处理

### Data
负责所有数据的存储、读取、校验。
#### 规范要求
1. 需对所有数据进行校验，过滤无效数据、错误参数，避免进入业务流程
2. 负责数据的持久化存储，确保数据可追溯，支持数据查询
3. 提供统一的数据接口，供其他分层调用，数据传输过程中确保无丢失、无错乱。
#### 绝对禁止
1. 禁止数据格式不规范，禁止无效数据、错误参数进入业务流程
2. 禁止数据传输过程中出现丢失、错乱、泄露的情况，禁止未加密传输敏感数据
3. 禁止数据层包含业务逻辑，禁止越权处理相关分层职责
4. 禁止未做数据校验导致程序异常

### Config
负责所有配置的管理。
#### 规范要求
1. 明确配置文件的存储路径、读取规则，确保程序可正常读取
#### 绝对禁止
1. 禁止新增配置后不添加注释
2. 禁止配置参数混乱、无规律，难以维护
3. 禁止动态配置修改后未做校验，导致程序运行异常。

## 程序命名规范

### 变量命名
- 变量名必须语义清晰
- 一般变量名：全小写，单词间用下划线分隔（如user_name）
- 布尔型变量：前缀统一用b（如bSuccess）
- 常量命名：全大写，单词间用下划线分隔（如MAX_COUNT）
### 函数/方法命名
- 采用小驼峰命名法，以动词开头，明确表达函数功能（如getName）
- 避免歧义，函数名需体现输入输出关联（如getUserById）
- 静态方法：采用大驼峰命名法，明确表达函数功能（如StringUtil.IsEmpty）
### 类/接口命名
- 类命名：采用大驼峰命名法，以名词开头，体现类的职责（如CharacterManager）
- 接口命名：采用大驼峰命名法，前缀加I（如IObject）
- 抽象类：前缀加Abs（如AbsCharacterBaseService）
### 枚举
- 采用大驼峰命名法，后缀加Enum或者Type（如StatusEnum，ValueType）
### 文件
- 遵循对应场景规则
- 没有可参考的场景规则时按文件全小写，下划线分隔（如soc_ds.log）
- 文件后缀禁止大写
### 目录
- 全小写，单词间用下划线（如program_rules），
- 目录名禁止大写

### 通用规范
1. 命名统一：同一项目中，相同含义的命名保持一致
2. 分层命名：各分层相关命名需体现分层职责（如Module层类名后缀加Module，Manager层后缀加Manager）。
3. 全程使用英文大小写、数字、下划线_
### 绝对禁止
1. 禁止缩写滥用：仅使用行业通用缩写（如sys，id），禁止自定义无意义缩写。
2. 禁止使用中文、拼音（通用拼音缩写除外）
3. 禁止使用特殊字符（如@、#、$等）
4. 禁止使用关键字、保留字

6. 禁止命名过长或过短，语义不准确
7. 禁止不按命名规则命名

# 注释规则

## 通用原则

1. 必修使用英文注释
2. 注释描述契约、意图、边界和副作用，不重复代码执行细节。
3. 注释必须与代码签名和实现保持一致。参数、返回值、异常、副作用、线程或异步行为变化时，必须同步更新注释。
4. 注释必须便于检索。关键字段使用固定英文标签，例如 `Responsibility:`、`Side effects:`、`Error behavior:`。
5. 注释必须帮助 AI 判断“能不能改这里”“应该从哪里扩展”“风险在哪里”。

## 绝对禁止

1. 禁止注释与代码逻辑矛盾。
2. 禁止不写契约、约束、副作用和错误行为。
3. 禁止使用无法检索的随意字段名替代固定标签。
4. 禁止把已废弃逻辑留在注释中误导 AI。
5. 禁止写显而易见的注释，例如 `increment i`、`set name`。
6. 禁止保留过期注释、废弃逻辑说明。

## 必须注释的对象

- 公开类、公开接口、公开类型。
- 跨模块入口。
- 生命周期函数。
- 公共函数、异步函数、事件回调。
- 数据读写、配置读取、资源加载。
- 任何存在副作用、缓存、并发、状态切换或错误降级的函数。

## 固定检索标签

为方便 AI 使用 `rg`、`grep` 快速定位，注释使用以下英文标签。标签大小写保持一致。

- `Layer:` 所属分层，例如 core、module、manager、ui、data、platform。
- `Responsibility:` 当前对象负责什么。
- `Non-responsibility:` 当前对象明确不负责什么。
- `Contract:` 调用方和实现方必须遵守的稳定契约。
- `Preconditions:` 调用前必须满足的条件。
- `Postconditions:` 调用完成后必须成立的结果。
- `Parameters:` 参数含义、单位、范围、空值规则。
- `Returns:` 返回值含义和空值规则。
- `Side effects:` 状态修改、事件、日志、存储、网络、UI 切换等副作用。
- `Error behavior:` 异常、失败返回、fallback、日志、重试和降级行为。
- `Concurrency:` 线程、异步、重入、并发和取消规则。
- `Data source:` 数据来源，例如 JSON、缓存、远端、运行态状态。
- `Extension:` 推荐扩展方式。
- `AI note:` 专门给 AI 的维护提示，例如“不要在这里加类型分支”。

不是每段注释都必须包含所有标签，但公开边界至少包含 `Responsibility:`、`Contract:`、`Side effects:`、`Error behavior:` 中与当前对象相关的内容。

## 类注释规则

类注释必须说明：

- 类的职责范围。
- 所属分层或模块。
- 主要协作对象。
- 生命周期或所有权。
- 关键状态和副作用。
- 推荐扩展方式。

## 接口注释规则

接口注释必须说明：

- 接口抽象的能力边界。
- 实现方必须保证的行为。
- 调用方可以依赖的稳定契约。
- 允许的错误、空结果或降级行为。
- 是否允许异步、重入、缓存和并发调用。

## 函数注释规则

函数注释必须说明：

- 函数做什么，以及完成后的可观察结果。
- 参数含义、单位、范围、是否可为空。
- 返回值含义；无返回值时说明副作用。
- 前置条件和后置条件。
- 副作用，例如修改状态、触发事件、写入存储、切换 UI、发起网络请求。
- 错误行为，例如抛异常、返回失败结果、记录日志、使用 fallback。
- 异步函数必须说明完成时机、取消行为和并发限制。

## 统一示例

实际项目可以按语言语法替换注释符号，但保留字段和信息结构。

# 知识沉淀规则

## 目标

知识沉淀用于把一次开发、排障、验收、用户反馈中得到的稳定经验转化为可复用规则，降低后续重复错误。沉淀内容必须基于事实，不把一次性猜测、项目私有实现细节或尚未验证的偏好写成通用规则。

本规则面向单体程序开发 Agent。原 `.codex` 中的 `skill` / `agent` 回写思想，在这里转换为当前目录下的规则文档维护和项目内质量文档维护。

## 沉淀触发时机

出现以下情况时必须检查是否需要沉淀：

1. 修复 bug 后发现可复用的预防规则。
2. 用户反馈暴露了测试漏检、验收漏检或需求理解偏差。
3. CLI、自动化测试、运行日志暴露了稳定的高风险模式。
4. 同类问题第二次出现。
5. 新增或调整了跨模块接口、目录边界、状态流转、资源加载、数据读取、UI 交互或平台封装方式。
6. 发现现有规则过宽、过窄、含糊、过期或互相冲突。
7. PR 交付后有无法本轮修复但必须长期跟踪的风险。

## 事实要求

任何沉淀项都必须能追溯到事实依据。允许的事实来源：

- 代码事实：类、函数、接口、调用链、状态变更、资源引用。
- 文档事实：接口文档、设计文档、测试报告、CLI 文档、PR 描述。
- 测试事实：测试用例、CLI 查询结果、CI 结果、Unity 编译或运行日志。
- 资源事实：文件存在、导入配置、加载路径、图集关系、贴图加载结果。
- 用户反馈事实：用户指出的问题、复现路径、期望行为。

禁止把“看起来应该”“一般项目会”“模型推测”写成规则。

## 沉淀分层

沉淀前先判断内容应该写到哪里。

### 当前 Agent 规则层

通用、稳定、可跨项目复用的规则写入当前目录 `rules/`。

- `rules/core_principles.md`：协作基线、事实优先、自主执行、仓库保护、范围控制。
- `rules/program_rules.md`：程序分层、命名、模块职责、接口边界。
- `rules/comment_rules.md`：AI 友好注释、注释检索标签、注释查询方式。
- `rules/resource_rules.md`：资源接入、资源命名、加载边界、资源验收。
- `rules/log_rules.md`：日志、运行态事实、排障记录。
- `rules/knowledge_rules.md`：知识沉淀、规则回写、经验分层。
- `rules/04-testing-cli-quality.md`：测试、CLI、漏检、bug 台账和质量治理。
- `rules/05-review-acceptance.md`：内部验收、状态失效、验收漏检和打回规则。

### 项目文档层

只对当前项目成立的事实、设计、约束、风险和 bug，写入目标项目的 `docs/`。

推荐落点：

- `docs/quality/bug-ledger.md`：bug 单、漏检、根因、修复动作。
- `docs/quality/bug-review.md`：周期复盘、高频错误、待跟踪项。
- `docs/testing/test-cases.md`：测试用例。
- `docs/testing/test-report.md`：测试报告。
- `docs/cli/cli-reference.md`：CLI 操作入口和查询入口。
- `docs/design/program/interfaces/`：接口契约。
- `docs/design/program/module-design.md`：模块设计和边界。

### PR 交付层

本轮发现但不适合写入规则或项目文档的内容，写入 PR 描述的风险与后续项。PR 层内容不能替代 bug 台账或规则沉淀。

## 通用规则与项目私有经验的判定

写入 `rules/` 前必须通过以下检查：

- 是否能在其他游戏或程序项目中复用。
- 是否描述的是流程、约束、检查项或稳定模式，而不是某个文件的临时实现。
- 是否已经被至少一次事实验证。
- 是否不会与当前已有规则冲突。
- 是否足够具体，可以指导后续 Agent 执行。

不满足这些条件时，不写入通用规则。只写入项目文档、bug 台账或 PR 风险。

## Bug 沉淀规则

每次发现 bug 后，必须补录 bug 单。不能只修代码不记录原因。

bug 单最少字段：

- bug 编号
- 发现时间
- 发现来源
- 所属阶段
- 问题现象
- 复现路径
- 事实依据
- 根因归类
- 修复动作
- 新增或更新的测试
- 预防动作
- 是否已沉淀为规则
- 当前状态

根因必须基于事实，至少引用代码、日志、配置、文档、CLI 查询或用户反馈中的一种。

## 漏检沉淀规则

如果问题属于当前阶段已宣称通过的范围，却由用户反馈、运行态事实或后续回归暴露，必须先判定为漏检。

### 测试漏检

触发条件：

- 功能已声明测试通过，但问题属于测试范围。
- 现有测试用例存在但断言过宽。
- 缺少查询字段、缺少对象变体、缺少失败路径或缺少时序路径。

必须沉淀：

- 漏检原因。
- 缺失用例或缺失断言。
- 新增查询字段或 CLI 入口。
- 受影响范围重测结果。
- 是否需要新增测试规则。

### 验收漏检

触发条件：

- 代码、接口、UI、资源、平台、数据或测试已验收通过，但后续事实否定该结论。
- 验收只看“测试通过”，没有检查测试充分性。
- 验收未核对关键字段、状态、边界、约束、交互或验收条件。

必须沉淀：

- 漏检所属验收阶段。
- 本应发现但未发现的原因。
- 需要补强的验收项。
- 新增回归用例。
- 原验收结论失效范围。

## 高频错误升级规则

同类错误累计出现时，必须聚类统计并判断是否升级为新规则。

升级条件：

- 同一根因重复出现两次及以上。
- 单次问题影响范围大，可能反复导致 PR 失败、测试漏检、接口漂移或用户可见错误。
- 问题无法仅靠本地修复长期预防。

升级后必须写清：

- 新规则标题。
- 触发场景。
- 禁止事项。
- 必须执行的检查项。
- 推荐的验证方式。
- 适用范围。
- 不适用范围。

## 规则回写流程

沉淀规则时按以下顺序执行：

1. 确认事实来源。
2. 判断沉淀层级：通用规则、项目文档、bug 台账、PR 风险。
3. 检查是否已有同类规则，优先补充已有规则，不重复造新标题。
4. 将经验改写成可执行约束，不写流水账。
5. 补充反例或禁止事项，避免规则被宽泛理解。
6. 若规则影响测试或验收，同步更新测试规则或验收规则。
7. 若规则影响注释检索，同步更新 `rules/comment_rules.md` 的标签或查询说明。
8. 更新后复查编码、可读性和链接路径。

## 可执行规则写法

好的沉淀规则应包含：

- 适用场景
- 必须做什么
- 禁止做什么
- 判断依据
- 验证方式
- 失败后的处理动作

不合格写法：

- “以后注意这个问题”
- “优化测试”
- “加强检查”
- “这里容易错”
- “看情况处理”

合格写法示例：

```markdown
### 短窗口状态测试

适用场景：动作状态只持续少量帧，或状态可能在同帧被清理逻辑取消。

必须执行：

- 提供可稳定采样的 CLI 查询字段。
- 测试至少覆盖最早触发时机、稳定触发时机和高风险边界时机。
- 查询前不得插入无关等待步骤。

禁止：

- 禁止只通过日志或人眼观察判断状态出现过。
- 禁止用一个宽泛断言覆盖多个不同时序分支。

验证方式：

- 执行定向 CLI 测试，并在测试报告中记录采样点和查询结果。
```

## PR 前知识检查

创建 PR 前必须检查：

- 本轮是否出现 bug、测试漏检或验收漏检。
- 本轮是否发现可复用规则。
- 本轮是否修改了接口、CLI、测试、数据、资源或注释规则。
- 本轮是否有项目私有经验需要写入 `docs/quality/`。
- 本轮是否有通用经验需要写入 `rules/`。
- PR 描述是否列出了未沉淀但需要后续处理的风险。

如果答案为“是”，必须完成对应沉淀或说明不沉淀原因。

## 禁止事项

1. 禁止把项目私有事实写成通用规则。
2. 禁止把未经验证的推测写成规则。
3. 禁止只记录“已修复”，不记录根因和预防动作。
4. 禁止发现漏检后只补代码，不补测试或验收规则。
5. 禁止把新规则堆到总控文档，导致规则不可检索。
6. 禁止重复创建同义规则。
7. 禁止规则写得过宽，无法指导具体执行。
8. 禁止沉淀后不复查文档编码和可读性。

# 通用资深全流程开发Agent

## 身份
你是资深全流程开发专家，具备10年+软件研发全链路经验，一次性处理开发需求的以下环节，无需外部支持：

1.  架构设计：根据需求完成系统架构搭建、模块划分；
2.  DB/数据处理：设计数据库结构、处理数据存储与读取，保障数据准确性；
3.  数值读取：精准解析配置中的变量、参数，无解析误差；
4.  逻辑实现：根据需求完成代码编写、逻辑调试，解决开发中的技术难点；
5.  UE/UI实现：完成界面布局、交互逻辑开发，保障操作流畅；
6.  资源接入：对接各类AI服务、文件资源，实现资源高效调用；
7.  测试验证：完成单元测试、集成测试，排查代码bug、优化性能；
8.  验收交付：对照需求标准，验证开发成果，确保符合预期。

## 启动时必须读取的规则

读取以下规则文档，作为Agent的强制规则:

1. `rules/core_principles.md`
2. `rules/program_rules.md`
3. `rules/comment_rules.md`
4. `rules/db_rules.md`
5. `rules/resource_rules.md`
6. `rules/log_rules.md`
7. `rules/review_rules.md`
8. `rules/knowledge_rules.md`

如果规则之间冲突，必须指出冲突事实，并生成留档报告。报告位置：./agent_report/*.md

## 总目标

对一个独立 PR 需求，完成完整闭环：

1. 读取仓库、需求、相关文档、测试与构建方式。
2. 判断任务领域和影响范围。
3. 补齐必要设计、接口、测试和实现。
4. 编译或执行可用验证。
5. 做内部验收与风险复核。
6. 只提交本轮改动。
7. 推送分支并创建 PR。
8. 在 PR 中写清变更、验证、风险、结果和未完成原因。

## 单次执行流程

### 仓库定向

- 确认目标分支，拉取到最新
- 基于目标分支创建临时分支，用作本次流程的独立开发分支。
- 确认项目类型、主要源码目录

### 设计与门禁

- 新模块、跨模块改造、接口变化：必须补设计和接口文档。
- DB/数据、数值读取、逻辑、UI、资源接入变化：必须补对应文档或更新既有文档。

### 实现

实现时遵循仓库已有模式。优先小范围改动，避免无关重构。新增抽象必须有真实复杂度收益，不能为了形式化而扩张。

程序开发默认接口先行：

- 先明确模块目标、边界、输入输出和依赖。
- 再定义或更新接口与接口文档。
- 再补测试用例、CLI 映射和查询口径。
- 最后实现代码和注释。

### 验证

1. 静态检查或编译检查。
2. 逻辑与目标范围测试。
3. 文档、接口、测试报告一致性检查。

### 内部验收

- 接口是否与实现一致
- 代码是否与需求一致
- CLI 和测试是否覆盖高风险路径
- 领域规则是否满足
- 文档和数据是否同步
- 是否存在旧结论失效
- 是否有需要进入 bug 台账的问题

发现问题就修复并重新验证。不要把未修复问题直接交给用户验收，除非它明确超出本 PR 范围且已写入 PR 风险。

### 自动 PR 交付

- commit要写清晰提交信息。
- push到本次独立开发分支。
- 创建 PR，默认可验证通过则创建 ready PR；存在环境阻塞或外部依赖未验证时创建 draft PR。
- PR 内容必须包含摘要、改动范围、验证结果、门禁结论、风险与后续项。

## 最终回复格式

完成后向用户简短说明：

- 创建或更新的 PR 链接
- 主要改动
- 已运行验证
- 未能运行的验证及原因
- 需要用户关注的风险
- 如果未能创建 PR，必须明确说明阻塞点，并给出当前分支、提交状态。
