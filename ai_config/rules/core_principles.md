# 核心原则

## 绝对禁止
- 禁止把需求停留在计划层。除非遇到真实阻塞，否则必须推进到 PR 交付。
- 禁止跳过事实读取直接实现。
- 禁止用主观猜测替代代码事实、资源事实、文档事实、运行态事实。
- 禁止把代码能跑当作验收通过。必须检查接口、测试、文档、领域规则和 PR 风险。

## 沟通规则

- 必须使用中文输出、中文文档和中文 PR 说明。
- 代码内注释按 `comment_rules.md` 约束，必须使用英文；仓库文档、面向用户的输出和 PR 说明必须使用中文。
- 先执行能自主完成的动作，再报告结果。
- 信息不足时，优先读仓库文档、代码、配置、测试、日志。
- 只有关键事实无法从仓库推断且会决定实现方向时才提前停止。并指出提前停止事实，生成留档报告。报告位置：本次 `ai-prompt.md` 所在运行目录下的 `agent-report.md`
- 面向用户的更新要说清读到了什么，思考过程，最终执行方案。
- 最终回复只保留对用户有用的信息，不输出冗长过程。

## 仓库规则

- Agent工作不与人类工作混用本地仓库空间。
- 执行前确保本地没有未提交改动。
- 当存在未提交改动时，还原改动到与远端目标分支一致。不受版控影响的文件除外。
- 如果不存在 Git 仓库，则继续完成本地可验证改动，但最终必须把 PR 交付阻塞写清。

## 范围控制

一个 PR 只处理一个独立程序需求。允许为了完成需求补齐必要的文档、测试、数据和资源配置，但不做无关重构。

1. 实现或修复用户明确要求。
2. 补齐本需求必须依赖的接口、数据、文档、测试。
3. 修复实现过程中暴露的同范围阻塞 bug。
4. 记录但不顺手修改无关问题。

## 事实优先

任何判断都要尽量绑定事实来源：

- 代码事实：文档、类、函数、调用链、配置读取、资源引用。
- 资源事实：文件存在、导入配置、加载路径、图集关系。
- 文档事实：已确认设计、接口文档、测试报告。
- 运行态事实：编译结果、日志、测试输出。

## 文件与编码

- 人工编辑的 Markdown、C#、UXML、USS 文档与脚本优先使用 `UTF-8 with BOM`。
- JSON、YAML、TOML、XML、meta 等机器配置文件使用 `UTF-8 无 BOM`。
- 含中文文件写入后要复查可读性。
- 新增文档优先放入仓库既有文档体系
- 非代码、配置类文件，优先使用中文名。
- 文档类内容必须使用中文文档名及中文文档内容。
